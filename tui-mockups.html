Tunnitup · interaction study

The proxy should feel simpler than the thing it hides.

Three directions for the first TUI. Each uses the same project, routes, and tunnel state so the difference is the mental model: monitor a running system, manage routes, or get guided from zero to public.

One obvious startEnter local ports, confirm editable suggestions, press one key.
Plain-language failuresSay which service failed and the exact next action.
Safe by defaultPreview public routes before opening the tunnel.
Details on demandKeep proxy vocabulary out of the primary workflow.

A · Command center

Optimized for the thing users do most after setup: start the tunnel, confirm routes are healthy, and glance at recent requests.

Best daily driver. Route editing is secondary, which keeps the running view calm but makes initial setup less discoverable.

B · Route workspace

Routes are the product. The main view makes path precedence, upstream health, and configuration changes explicit.

Best for learning and control. It teaches the abstraction well, but feels more like infrastructure tooling during everyday use.

C · Guided launch

A first-run flow that probes only the ports the user provides, proposes editable routes, checks ngrok, and previews exactly what becomes public.

Best onboarding. It removes blank-page setup, but should hand off to a compact running view instead of becoming the permanent shell.

Chosen direction

C for first run, A thereafter

Ask for ports, probe only those services, and offer editable path suggestions. After the user confirms the public mapping, launch directly into the command center. Borrow B’s route detail screen only for editing.
